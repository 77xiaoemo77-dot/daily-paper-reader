## 摘要
约束编程（Constraint Programming, CP）是一种解决组合问题的强大范式，但将自然语言问题描述转换为可执行模型仍然是一个显著的瓶颈。尽管大型语言模型（Large Language Models, LLMs）在自动化这一转换过程中展现出潜力，但在测试阶段缺乏“预言机”验证时，它们常常难以避免细微的语义错误。为此，我们提出了 CP-SynC（利用合成检查器的约束编程建模，Constraint Programming modeling with Synthesized Checkers），这是一种在 MiniZinc 中进行零样本约束建模的多智能体工作流程。CP-SynC 协调建模智能体生成并优化候选模型，同时让验证智能体合成语义检查器，为语义正确性提供反馈。为了缓解单个 LLM 输出中固有的噪声，CP-SynC 并行探索多条建模轨迹，并通过选择智能体进行多智能体证据汇聚以选定最终模型。基于包含 100 个 CP 问题的基准测试实验表明，CP-SynC 在 MiniZinc 建模中显著优于现有基线方法。

## Abstract
Constraint Programming (CP) is a powerful paradigm for solving combinatorial problems, yet translating natural language problem descriptions into executable models remains a significant bottleneck. While Large Language Models (LLMs) show promise in automating this translation, they often struggle with subtle semantic errors in the absence of oracle validation at test time. To address this, we introduce CP-SynC (Constraint Programming modeling with Synthesized Checkers), a multi-agent workflow for zero-shot constraint modeling in MiniZinc. CP-SynC coordinates modeling agents that generate and refine candidate models and validation agents that synthesize semantic checkers to provide feedback on semantic correctness. To mitigate noise inherent in individual LLM outputs, CP-SynC explores multiple modeling trajectories in parallel and employs selection agents to select the final model via multi-agent evidence aggregation. Extensive experiments on a benchmark of 100 CP problems show that CP-SynC substantially outperforms existing baselines in MiniZinc modeling.

## 二、方法论：CP-SynC 多智能体框架

### 1. 核心思想  
CP-SynC（Constraint Programming modeling with Synthesized Checkers）是一个**多智能体工作流**，通过“建模—验证—选择”三类代理间的协作，实现对自然语言问题自动生成 MiniZinc 模型的零样本建模。

### 2. 核心组成  
- **建模智能体（Modeling Agents）**  
  - 输出多个候选 MiniZinc 模型；  
  - 根据错误反馈进行“自我修复”（self-refinement）；  
  - 支持语法级（G1–G3）与语义级（G4）多阶段检查反馈。
- **验证智能体（Validation Agents）**  
  - 基于问题描述自动生成“语义检查器（semantic checkers）”；  
  - 检查候选模型输出是否符合问题约束，如变量一致性和逻辑正确。  
  - 输出形式为 `(pass/fail, feedback)`。
- **选择智能体（Selection Agents）**  
  - 聚合建模结果与验证反馈，进行多视角评估；  
  - 采用多数投票策略（majority vote）选择最终模型。  

### 3. 算法流程概述（对应图 1）
1. **初始化阶段**：为每个角色生成 K 个独立代理。  
2. **多候选生成**：建模智能体生成 K 个候选 MiniZinc 模型。  
3. **语义检查器合成**：验证智能体生成 K 个 Python 语义检查函数。  
4. **多级检查与自我优化**：模型通过四阶段检查：
   - G1：语法正确性  
   - G2：求解状态（SAT/UNSAT）  
   - G3：输出格式一致性  
   - G4：语义检查（多数通过即验收）  
   若失败，则调用自我修复循环（最多 r 次）。  
5. **模型选择与重启机制**：若全体候选不通过，重新生成描述并重复流程（最多 R 次重启）。  
6. **多轨迹并行探索**：同时探索多个独立轨迹，以提升语义收敛概率。  

### 4. 技术细节  
- 引入两类“生成多样性策略”：  
  - **多样化提示采样（Prompt Diversification）**：变换问题描述；  
  - **温度采样（Temperature Sampling）**：非零温度随机采样。  
- 融合**语义检查器验证信息聚合**与**多数投票**策略，确保系统鲁棒性。

## 六、主要实验结果与结论

- **总体性能**  
  CP-SynC 在所有评测的 LLM 上均显著优于 2S 与 CP-Bench：
  - 以 GPT-4o 为例：**65.3 → 78.0 (+12.7%)**；
  - 以 DeepSeek-V3 为例：**56.7 → 77.7 (+21%)**；
  - 小模型 Gemini 提升最显著（19 → 67 含 3.5 倍性能提升）。  
- **解释**：  
  性能提升源自于：
  1. 多轨迹生成避免单一错误路线；  
  2. 合成检查器提供语义反馈；  
  3. 多智能体投票增强鲁棒性。  
- **最强系统**：Claude-Sonnet-4 + CP-SynC 组合取得最高总分（86.3 SA）。
