## 摘要
本文介绍了 ORPilot，这是一种开源的代理式人工智能系统，能够将现实世界中的商业问题转换为可直接求解的优化模型。与学术界的 LLM-for-OR 工具（假定问题规范清晰并带有预格式化的内嵌数据）不同，ORPilot 专为生产环境设计，能够处理含糊的描述、大规模原始运营数据，并支持跨求解器后端的可移植性。该系统引入了四个创新组件：(1) 对话式访谈代理，用于引出完整的问题规范；(2) 数据收集代理，可独立于提示获取数据；(3) 参数计算代理，用于连接原始表格数据与模型就绪参数；(4) 求解器无关的中间表示（IR），可实现确定性的、零 LLM 调用的再编译，以支持 Gurobi、CPLEX、PuLP、Pyomo 或 OR-Tools 等求解器。此外，自校正的重试循环利用求解器回溯信息进行针对性修复。ORPilot 是首次尝试针对生产级商业问题而非教材中的运筹学（OR）案例。实证评估表明，在真实业务问题上的表现令人鼓舞。与传统学术基准测试 IndustryOR、NL4OPT 和 NLP4LP 比较时，ORPilot 在 IndustryOR 基准上的准确率优于当前先进工具，并在 NL4OPT 和 NLP4LP 上表现相当。

## Abstract
This paper presents ORPilot, an open-source agentic AI system that translates real-world business problems into solver-ready optimization models. Unlike academic LLM-for-OR tools that assume clean problem specifications with preformatted inline data, ORPilot is designed for production conditions: ambiguous descriptions, large-scale raw operational data, and the need for portability across solver backends. The system introduces four novel components: (1) a conversational interview agent to elicit complete problem specifications, (2) a data collection agent that retrieves data independently of prompts, (3) a parameter computation agent to bridge raw tabular data and model-ready parameters, and (4) a solver-agnostic Intermediate Representation (IR) for deterministic, zero-LLM-call recompilation to Gurobi, CPLEX, PuLP, Pyomo, or OR-Tools solvers. Additionally, self-correcting retry loops utilize solver tracebacks for targeted repairs. ORPilot represents the first attempt to target production-level business problems rather than textbook operations research (OR) cases. Evaluation on real-world problems demonstrates promising results. When tested against traditional academic benchmarks: IndustryOR, NL4OPT and NLP4LP, ORPilot outperformed state-of-the-art tools in accuracy on the IndustryOR benchmark and delivered comparable performance on NL4OPT and NLP4LP.

## 一、研究背景与核心问题

- **背景**：  
  运筹学（Operations Research, OR）长期被用于物流、供应链、调度、金融等决策优化场景。尽管求解器技术（如 Gurobi、CPLEX）已非常成熟，但现实企业仍难以直接构建和部署优化模型。主要瓶颈在于：
  - 企业用户缺乏数学规划专业知识；
  - 问题描述往往含糊不清、存在遗漏或自相矛盾；
  - 数据通常存于业务系统或原始表格，而非结构化输入。

- **核心问题**：  
  现有的 LLM-for-OR（利用大语言模型生成优化模型）系统多在理想化的学术环境下测试，只能处理“干净的、完整的教材式问题”，无法应对真实生产条件——例如不完备的描述、大规模原始数据以及跨求解器的复用需求。

- **研究动机**：  
  作者提出 **ORPilot**，旨在构建一个面向生产部署的“智能体化 LLM-OR建模系统”，可从模糊的业务描述与原始数据自动生成可复用的优化模型，实现真正的业务级自动化建模。

## 二、方法论与系统架构

### 核心思想
- 将 LLM 融入到一个多智能体（multi-agent）管线中，让不同角色负责问题澄清、数据解析、参数计算和求解器生成。
- 通过构造 **求解器无关的中间表示（Intermediate Representation, IR）** 实现模型的复现与跨求解器兼容，无需再次调用 LLM。

### 四个关键组件

1. **访谈代理（Interview Agent）**  
   - 与用户交互，主动提问补全缺失信息，直到目标、变量、约束与参数完整、无歧义。
   - 模拟人类 OR 顾问的工作流程：先澄清问题再建模。
   - 在结束时提供结构化摘要（目标函数、决策变量、约束、参数索引）。

2. **数据收集代理（Data Collection Agent）**  
   - 指导用户提供必要的原始数据（CSV 格式），明确每类数据的结构与类型。
   - 验证数据完整性和一致性，确保建模前的数据准备充分。

3. **参数计算代理（Parameter Computation Agent）**  
   - 自动识别需要计算或汇总的派生参数（如距离矩阵、聚合需求、BigM 等）。
   - 生成并执行 Python 脚本，将原始数据转换为模型就绪参数。
   - 这是首次在 LLM-for-OR 领域显式处理“从原始表到参数”的转换层。

4. **中间表示（IR）与编译器**  
   - 以结构化 JSON 表达优化模型的五要素（集合、参数、变量、目标、约束）。
   - 经验证后，可确定性再编译为任何支持的求解器代码（Gurobi、CPLEX、PuLP、Pyomo、OR-Tools）。
   - 后续修改（what-if分析）可在 IR 层完成，无需重新调用 LLM。

5. **自校正重试循环（Retry Loop）**  
   - 任何求解器错误、不可行或非有界状态，均反馈给 LLM进行有针对性修复。
   - 一般 1–2 次重试即可修复，大幅提升稳定性。

6. **报告代理（Reporter Agent）**  
   - 将模型结果转化为可读的业务报告，解释关键决策与数据洞察。

### 管线概览（Figure 1 对应）
- 访谈 → 数据收集 → 参数计算 → 代码生成 → 执行与修复 → 报告与IR持久化  
- 蓝色节点为 LLM 驱动步骤；橙色节点为确定性处理步骤。

## 三、实验设计

  - 验证系统各组件独立工作流程。

  - 运输任务与承运商匹配，含 3200货运与400承运商的二值决策。

## 六、主要结论与发现

- ORPilot首次实现从自然语言到可部署优化模型的闭环系统，并在大规模工业场景中稳定运行。
- 提出的四个代理模块有效弥补实际业务中的信息不完备与数据格式不匹配问题。
- IR机制显著提升模型的**可复现性、可移植性与求解器独立性**。
- 在工业型数据集（IndustryOR）上取得比现有系统更高的准确率，验证其面向生产环境的优势。
- 实验报告表明，即使在上千万变量的 LP 问题上，也能成功求解，显示出良好的系统效率。
